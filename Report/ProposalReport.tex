\begin{abstract}
In this report, it is proposed to utilize a reinforcement learning program in conjunction with one or more basic path planning algorithms to efficiently optimize a simulated turtle-bot travel path in a 2D environment based on the manipulation of several key parameters with the performance measured against path length, smoothness, travel time, velocity, and clearance.
\end{abstract}

\section{Introduction}
In the industrial environment of AMRs, path planners are, in general, deployed en-mass and are not tailored to the customers needs or the warehouse environment in which they are deployed. This results in good, but not excellent performance, and while easier to maintain on the backend, can limit customer output on the front end. These general deployments often cast a wide net and aim to operated on the `safe side', whether that be wide, nonadjustable clearance, paths around high traffic areas, or creating lanes that avoid foot traffic altogether. In reality, each customer has different needs, and outside of required safety standards, may have certain workplace standards that could offer better performance if utilized properly to tune the AMR deployment. It is obvious that hand tuning any number of parameters for each individual deployment site is unreasonable and the time spend on it would far outweigh any long-term benefits.
To reduce the amount of work input needed to tune these parameters, it is proposed that a reinforcement learning (RL) algorithm is deployed for each site and the parameters are adjusted with direct input of the customer to optimize the AMR deployment for their specific site while prioritizing their specific needs. The path evaluation metrics utilized in this simulation will be based on the following criteria: 
\begin{itemize}
    \item Safety (clearance from obstacles)
    \item Accuracy (success rate in reaching the goal, amount of deviation from global path plan, etc.)
    \item Efficiency (total time and distance to goal)
    \item Smoothness (heading angle variance, velocity variance, etc.)
\end{itemize}
    \subsection{Deliverables}
    At the end of the semester, the team will have a functional program that runs a 2D simulation utilizing a Turtlebot which needs to navigate through a crowded warehouse-like environment utilizing one or multiple path planners in conjunction with an RL algorithm that will automatically tune the path and path parameters to optimize the path of travel to align with the request. In this simulation, getting from point A to point B the fastest will not be the only measure of success. The RL algorithm should be able to take in parameter adjustments to the desired path, acceptable smoothness, desired clearance, and zone restrictions in order to create an optimal plan for the turtlebot.
\section{Project Progress}
    \subsection{Related Work}
    To begin, research was conducted to establish any relatable prior work; keywords under study included: path / trajectory planner, reinforcement learning, ROS2, and MPPI. Below are some of the most recent or relevant papers uncovered that either analyzed a similar problem, or utilized part of the proposed approach in a different environment.
        
        \subsubsection{Comparative Analysis of Local Trajectory Planning Algorithms in ROS2 \cite{b1}}        
        In this paper, Elbouhy et. al designed a study that consisted of both a simulation and real world test of a small mobile robot utilizing ROS2 and three different path planners. The path planners were tested in both environments with static and dynamic obstacles and the results were then evaluated to compare algorithms. The study analyzed a DWB (Modified Dynamic Window Approach), MPPI (Model Predictive Path Integral), and RPP (Improved Pure Pursuit) planner and used the following criteria to determine path quality: time, distance, smoothness, minimum clearance, runs completed, and number of recoveries (if obstacle contact is made). After running 10 runs of each planner in both the simulation and real environments, it was clear that MPPI was the fastest but least smooth, DWB operated with the least clearance, and RPP took the most time and traveled the longest distance.

        As this relates to this project, this clearly demonstrates that, as expected in some cases, different planners have different strengths and weaknesses. One of the potential goals for this project is to utilize multiple path planners in concert with the RL algorithm and the research conducted in this paper shows that an approach like that could, in some circumstances, optimize a path. It also shows that if certain criteria are considered higher value, then the RL could favor those. For example, if the environment is close quarters and completely static, DWB could be favored for it's low clearance and shorter distance traveled.
        
        \subsubsection{Local Path Planning Algorithm of Mobile Robot by Modified Optimized Parameters Method (2010) \cite{b2}}        
        In this paper, Li et. al studied the Dynamic Window Approach (DWA) algorithm and adjusted specific evaluation parameters in an effort to optimize the path planning sequence. DWA is a local path planner that, in this case, works in concert with a global planner. DWA measures and simulates velocity values for the robot over a specific period of time, then with that, it generates and evaluates the paths created and determines the optimal choice. The evaluation function utilized in this paper looked at the heading, or the difference in angle between the current and goal state, the distance between the trajectory and nearest obstacle, and the speed of the robot, along with weighted modifiers of each ($\alpha$, $\beta$ and $\gamma$).
        A simple environment was set up in both a simulation and real-world and the tuning modifiers were adjusted as follows for 3 separate preliminary trials: (0.05, 0.10, 0.10), (0.10, 0.20, 0.20), (0.15, 0.30, 0.30). And it was determined that $\alpha$ had a minimal effect on optimizing the path and was set to 0.10 for the rest of the experiment. $\beta$ and $\gamma$ were then tuned between 0.05 and 0.4 in a series of 64 combinations to optimize both at $\beta$ = 0.04 and $\gamma$ = 0.36. 

        In the context of this project, the interest lies in the tuning equations itself. With the goal of this project being to implement a RL-supported algorithm for the path planning, the tuning equation above provides not only quality reference for this project, but also acts as a confident proof-of-concept in that it is known that path planning algorithms can be tuned and optimized on a limited number of parameters and the results are statistically significant and positive.

        \subsubsection{RL-Driven MPPI: Accelerating Online Control Laws Calculation With Offline Policy \cite{b3}}
        In this paper, Qu et. al. present a method of training RL policy offline to accelerate the convergence for the online control law calculation of MPPI in an unmanned aerial vehicle (UAV) application. The method they proposed involved utilizing the Distributional Soft Actor-Critic (DSAC) algorithm to have an RL agent train a pair of neural networks to model distributions for the control costs and rollout guiding policy used directly within the online MPPI control. So in this way, the RL agent improved MPPI by providing better ``first guesses'' than standard MPPI. The results of their experiments showed that their method converged significantly faster, produced smoother trajectories, and provided higher success rate in reaching the goal than the standard MPPI control law. Additionally, as part of the experiments in this paper, they also trained a separate RL agent to control the UAV directly (``direct RL control'') and compared the results to the RL-driven MPPI approach. While the RL agent performed better than the standard MPPI, the RL-driven MPPI approach still outperformed the direct RL control approach in terms of path smoothness and success rate in reaching the goal.

        The guiding principle of this paper translates directly to this project, as RL is being used to enhance the strengths of existing local planners instead of replacing them. In fact, results from this paper suggest that the direct RL control approach may not be the most optimal approach for a given path planning application. By leveraging the strengths of both RL and traditional path planning algorithms, this project aims to create a more efficient and adaptable path planning solution for mobile robots in complex environments.

        \subsubsection{Accelerating Deep Reinforcement Learning for Mobile Robot Path Planning with Curriculum Learning and Reward Shaping\cite{b4}}
        In this research paper, Ting et. al. present a modified Proximal Policy Optimization (PPO) in conjunction with a deep reinforcement learning (DRL) algorithm in order to reduce the time needed  to move an agent through an environment. Traditional DRL can require millions of episodes in order to converge on a single action. To reduce the number of episodes required and thus the calculation time for each move, they proposed a distance-based weighted dense reward function. The reward function they proposed is seen below where $R_g$ is the reward when the goal is reached, $R_c$ is the collision penalty, $R_s$ is a small step penalty, and $R_sh$ is a distance shaping reward.
        \[R = R_g+R_c+R_s+R_sh\]
        The critical part of the equation is $R_sh$, which utilized the euclidean distance to the goal to provide a higher value reward for better steps. For the RL algorithm we are implementing, we intend to use a similar reward function to reduce the learning time needed for each run or environment we simulate. 

        \subsubsection{A Hybrid Approach to Indoor Social Navigation: Integrating Reactive Local Planning and Proactive Global Planning \cite{b9}}

        In this paper, Debnath et al. propose a modular navigation framework for  indoor social navigation that combines a classical global planner with a 
        deep reinforcement learning (DRL) based local planner. The global planner generates waypoints and assigns Gaussian soft costs around detected pedestrians 
        to anticipate their future positions, while the trained local planner dynamically follows these waypoints while avoiding imminent collisions in real time. 
        The reward function used to train the local planner includes eight terms covering goal reaching, collision penalties, waypoint tracking progress, pedestrian 
        proximity avoidance, and a small per-timestep penalty to encourage efficiency. Through experiments across both cooperative and uncooperative pedestrian scenarios, their hybrid system outperformed both a traditional 
        DWA planner and a standalone RL baseline, with the RL-augmented system achieving a notably lower collision rate. 

        This paper is directly relevant to the dynamic parameter tuning approach adopted in this project. Their reactive local RL planner continuously 
        adjusts navigation behavior based on local observations (including proximity to obstacles, heading toward waypoints, and distance to goal), at every timestep 
        during navigation. This is very similar to what the proposed implementation in this project is, where the RL agent observes local state and adjusts planner parameters throughout the 
        navigation process rather than making a single configuration decision upfront. Their reward structure combines dense per-step terms with sparse goal and collision signals and their result that a learning-based local planner substantially outperformed fixed-parameter DWA confirms the premise of POT-r: that static, hand-tuned 
        planner parameters can miss out on meaningful performance beneifts. By leveraging an RL agent to dynamically tune these parameters, this project aims to achieve similar performance gains in path quality and safety across a range of warehouse environments.

        \subsubsection{Deep Reinforcement Learning with PPO for Autonomous Mobile Robot Navigation Using ROS 2 Framework \cite{b10}}

        In this paper, Rana and Kaveendran implement a PPO-based deep reinforcement learning agent to train a TurtleBot3 navigation policy from scratch within a ROS2 environment. The agent observes the robot's 
        surroundings through a LiDAR scan vector and directly outputs normalized linear and angular velocity commands at each timestep, making it the sole navigation controller rather than an augmentation of an 
        existing planner. Training was conducted using a minimal two-term reward function, including a small positive reward for forward progress and a large 
        negative penalty for collisions. This took roughly 4 hours using Stable-Baselines3's PPO implementation integrated with a custom ROS2 Gymnasium environment. The trained policy resulted in better performance, and behavior improvements such as 
        slowing near tight spaces. 
    
        This paper is particularly relevant as a validation of a similar implementation being used in this project: ROS2, Gymnasium, PPO, and a TurtleBot platform. The key 
        distinction is that their RL agent replaces the planner entirely by outputting direct velocity commands, whereas this project's agent acts as a meta-controller. Their training time 
        was approximately four hours for a significantly harder problem (the agent must learn full navigation from raw sensor data), providing reassurance that the team's proposed approach is viable.

        \subsubsection{Other Noteworthy Resources}
        In addition to the papers described above, there were a number of other papers that were reviewed which provided supplemental information for shaping the project. \cite{b5} and \cite{b6} provided metrics and insights on planner path evaluation which we plan to utilize for our own evaluation metrics. \cite{b7} and \cite{b8} demonstrated more examples which utilized RL within some sort of path planning framework for a mobile robot (including both direct and indirect RL control).

        \subsubsection{Gaps in Literature}
        While there are several papers which demonstrate the use of RL in path planning, no papers were identified which utilized RL to configure the parameters for a given local path planning algorithm. This is a critical gap in the literature, as it is well established that the performance of local path planners can be highly sensitive to their parameter settings, and that optimal settings can vary significantly across different environments and use cases. Meanwhile, some of the examples utilizing direct RL control are not very extensible as these methods tend to only train the RL agent to operate in a specific environment; and re-training for more complex environments can be very costly. By leveraging RL to automatically tune these parameters, our project aims to fill this gap and demonstrate a novel approach to optimizing path planning performance in a way that is adaptable to the specific needs of different deployment sites.

    \subsection{Simulation Environment}
    \subsubsection{Implemented Stages}
    The simulation environment was designed to be cross-platform and reproducible, leveraging Docker to encapsulate all dependencies and ensure consistent behavior across the team's Mac, Windows, and Linux development machines. The environment is structured as a two-layer Docker image: a base capstone\_ros\_layer image that installs all system-level dependencies (ROS2, Nav2, TurtleBot4 description packages, and visualization tools), and a top-level rbe\_capstone image that layers in the project's ROS2 workspace. Separating dependency installation from application code allows the more expensive base layer to be rebuilt infrequently while code iteration remains fast.

    The simulation runs on ROS2 Jazzy with the CycloneDDS middleware layer for inter-node communication. The robot model used is the TurtleBot4, whose URDF is loaded via \texttt{xacro} and published to the \texttt{/robot\_description} topic by the \texttt{robot\_state\_publisher} node. Navigation is performed in a 2D occupancy grid environment using the Nav2 stack, which includes \texttt{nav2\_map\_server} for loading pre-built maps and \texttt{nav2\_lifecycle\_manager} for managing the lifecycle of navigation nodes. Maps are stored as paired \texttt{.pgm} and \texttt{.yaml} files and are selectable at launch time, enabling the team to quickly swap between different warehouse environments as the project progresses.

    Visualization is handled by RViz2, which renders the robot model, occupancy map, and planned paths. On macOS, because the simulation runs headlessly inside a Docker container, a virtual framebuffer (Xvfb) and VNC server are used to forward the graphical interface to the host machine, accessible on port 5900 or via a browser-based client on port 6080. A single parameterized launch file, \texttt{view\_robot.launch.py}, brings up all required nodes---the robot state publisher, static TF broadcasters, map server, lifecycle manager, and RViz2---making it straightforward to stand up the full environment with a single command. For the purpose of 
    this project, RVIZ2 is chosen over Gazebo for visualization because it is more lightweight and the scope of this project does not require a full physics simulation. Gazebo will be consistently slower 
    in launching and running episodes, which is not ideal for iterative tests and tuning of the RL algorithm. RViz2 provides sufficient visualization capabilities for the 2D occupancy grid and planned paths while allowing for faster iteration cycles.

    \subsubsection{Planned Simulation Development}
    The next phase of simulation development focuses on transitioning the environment from static visualization to an active, navigable system capable of supporting the reinforcement learning pipeline. This involves three parallel workstreams: navigation integration, trajectory evaluation, and environment expansion.

    The immediate priority is wiring in an active odometry source and configuring the Nav2 local planners so the TurtleBot4 can receive and execute navigation goals within the occupancy grid. Both DWB and MPPI will be brought up as configurable Nav2 plugins, with their tunable parameters exposed via the ROS2 parameter server. This is a prerequisite for the RL component, as the agent must be able to set planner parameters and trigger navigation runs programmatically.

    Once the robot can navigate, a trajectory evaluation module will be implemented to compute the path quality metrics defined in Section I: path length, path smoothness (heading angle variance), travel time, average velocity, and minimum obstacle clearance. These metrics will be computed from the \texttt{/odom}, \texttt{/cmd\_vel}, and \texttt{/local\_costmap} topics published during each navigation run. This module will be implemented as a standalone ROS2 node that subscribes to the relevant topics during an episode and produces a structured result upon goal completion or timeout.

    To ensure the RL agent generalizes across site layouts rather than overfitting to a single map, multiple warehouse-style occupancy grid maps will be introduced. Maps will vary along key structural dimensions---corridor width, obstacle density, and open floor area---to expose the agent to the range of layouts described in the project motivation. Together, these components will complete the simulation layer and allow the reinforcement learning environment to be built directly on top of it.


    \section{Next Steps}

    \subsection{Reinforcement Learning}
    The RL component of the project will be developed within the simulation environment, with the goal of having a trainable RL agent that can tune planner parameters directly through the Nav2 stack. The RL agent will be implemented using the PPO algorithm, but there were two design options explored for RL architecture of the states, actions, and rewards. It's worth noting that states, actions, and rewards for each design are not fully fleshed out due to the iterative nature of RL development and the tinkering that's often involved within machine learning workflows; but the general structure of each design is described below along with the pros and cons of each approach. 

    \subsubsection{One-Time Parameter Tuning during Initialization}
    With this approach, the RL agent is only responsible for configuring the local planner's parameters one time before the robot begins navigating (e.g. at the start of each episode):
    \begin{itemize}
        \item State would need to include 2D map, start pose, goal pose, and potentially the path generated from the global planner.
        \item The action space would include presets for the local planner parameters.
        \item The reward function would be derived from the path quality metrics defined in Section I, with the goal of maximizing path efficiency and smoothness while minimizing travel time and clearance. 
    \end{itemize}

    This approach composes of one state, one action, and one reward for each episode; so it's relatively straightforward to implement. However, since the one state will need to include the 2D map, the dimensionality of the state space may be very high (e.g. needing to include dimensions for coordinates of all obstacles in the map or pixel and color channel matrices for a pixel based representation of the map). To add to the high dimensionality, since there is only one state/action/reward per episode, the RL agent may require an extremely large number of episodes to learn anything meaningful due to the limited amount of feedback it receives on the quality of its actions per episode.

    \subsubsection{Dynamic Parameter Tuning during Navigation}
    With this approach, the RL agent would be responsible for dynamically tuning the local planner's parameters throughout the navigation process:
    \begin{itemize}
        \item State would need to include the robot's current pose, velocity, distance from the goal, and information derived from any on-board sensor data (e.g. sampled LiDAR segment returns). The state vector could also be augmented to include prior poses or future waypoints from the global planner to provide more context for decision making.
        \item Actions could either include presets for the local planner parameters or adjustments to the parameters (e.g. increase clearance by 0.1m, decrease path smoothness by 0.05, etc.).
        \item The reward function would resemble more standard direct RL control reward functions with a large positive reward for reaching the goal, a large negative reward for collisions, and small negative rewards for all other movements to encourage finding the shortest path to the goal. The reward function could also be expanded to include some of the metrics described in Section I for path quality, such as applying a small negative reward for high path curvature or large changes in velocity to encourage smoother paths.
    \end{itemize}

    This is a more ``hands on'' approach as the RL agent is more involved during the navigation process than the one-time tuning approach; so this approach is likely to be more complex to implement. However, since the state vector is a lower dimensionality and more closely tied to interactions with the environment, the RL agent will have more feedback during each episode to learn from and may converge to an optimal policy more quickly than the one-time tuning approach.

    \subsubsection{Current Decision}
    Given the time constraints for the project, the dynamic parameter tuning approach will be used going forward. While the one-time tuning approach may seem simpler to implement on the surface due to its relatively ``hands off'' nature, the lack of feedback combined with potentially high dimensionality of the state space will make RL training too expensive. If anything, the dynamic tuning's ``hands on'' nature should make it more malleable from an RL training perspective, as it provides the agent the opportunity to learn from each time step during navigation within a single episode. Regardless of the method used to translate a static map into a state vector (as this is a separate complicated problem in and of itself), the state vector of the dynamic tuning approach is much more similar to the inputs of the local planner than the state vector of the one-time tuning approach. This should help facilitate easier learning and decision making for the RL agent (in terms of function approximation) since there's already a strongly established correlation between parameters in the state vector and the performance of the local planner that the RL agent is trying to optimize. Additionally, the actions for the dynamic tuning approach may be more ``forgiving'' than the action for the one-time tuning approach, which should lead to more stable results. For example, with the one-time tuning approach, the robot is stuck with the same parameters throughout navigation until it reaches the goal or collides with an obstacle; whereas the dynamic tuning approach provides some opportunity for the RL agent to walk back any potential missteps or overcorrections that it makes to the planner parameters before it's too late. Of course, being stuck with the same parameters for the entire episode with the one-time tuning approach isn't any different than running with an RL free planning solution that's been manually tuned; but that just means the available actions (combinations of parameter configurations) for the one-time tuning approach need to be just as carefully curated as the manually tuned parameters for the RL free solution. In other words, the one-time tuning approach still requires meticulous parameter tuning while the dynamic tuning approach allows for some self-correction.

    In fact, the one-time tuning approach can be boiled down to a single function approximation problem. Instead of making a ``decision'' about what preset of values work from a discrete set of choices for a given state, it makes more sense to reframe the problem as a regression problem that takes the state vector as an input and outputs an optimal planner parameter vector. Training this sort of function approximation model through more direct deep learning methods would likely be more efficient than training an RL agent to do something similar. Instead of having to learn through experience, a set of runs could be made offline to create a dataset of state vectors, planner parameter vectors, and results (e.g. a trace of the trajectory taken or the path quality metrics). This dataset could then be fed into a supervised learning model to learn mappings between planner parameters and the resulting path quality for a given state. Once the mappings have been established through the trained model, a separate model could learn to generate optimized planner parameters for a given state by feeding its outputs directly into the already trained model to evaluate the resulting path quality (like an autoencoder with a frozen decoder). This would be a much more direct way to learn the optimal planner parameters for a given state without having to go through the trial and error process of RL training. The dynamic tuning approach, on the other hand, is more naturally suited to an RL framework since it involves making sequential decisions over time with feedback from the environment, which is what RL is designed for. If anything, the one-time tuning approach could be a follow-up project implemented on top of the dynamic tuning approach to serve as the initial parameter configuration. This would give the RL agent a better starting point to work with and potentially improve the final path quality or speed up convergence.
